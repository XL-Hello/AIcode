# Linux / UART 预备知识

这份文档不是命令速查，而是给嵌入式 Android 串口开发的入门预备知识。
目标是先弄清楚“UART 是什么、Linux 里怎么表示、开发时最容易出什么问题”，再去看后续的操作命令和调试方法。

## 1. 你需要先建立的整体认知

在嵌入式 Android 项目里，串口通常会出现在这些场景中：

- 开机日志输出
- Bootloader / U-Boot 调试
- Kernel 调试
- 系统启动后的调试控制台
- 外设通信，例如 GPS、蓝牙模块、扫码枪、MCU、传感器

你可以把 UART 理解成“设备之间最基础的一种文字通信方式”。  
它不像以太网那样复杂，也不像 USB 那样有完整协议栈，核心目标就是：

- 把字节可靠地从 A 发到 B
- 双方提前约定好通信参数
- 收发双方使用同一套规则

---

## 2. UART 是什么

UART 是 `Universal Asynchronous Receiver/Transmitter` 的缩写，中文常译为“通用异步收发器”。

这里要注意两个关键词：

- `Asynchronous`：异步，表示通信双方不共享时钟线
- `Receiver/Transmitter`：接收器/发送器，说明它负责把并行数据和串行数据互相转换

对初学者来说，可以先记住一句话：

> UART 是一种最常见的串行通信硬件接口，常用于嵌入式设备和调试口。

严格来说，大家日常说的“串口”很多时候其实是在泛指 UART 接口。

---

## 3. UART、TTL、RS-232、USB 转串口的区别

初学时最容易混淆的就是这几个概念。

### 3.1 UART

UART 是通信控制器和通信方式的名字，关注的是“怎么发、怎么收”。

### 3.2 TTL 串口

很多开发板上的串口引脚实际是 TTL 电平串口。

常见特征：

- 电平通常是 `3.3V` 或 `1.8V`
- 直接接 MCU、SoC、模块
- 不能直接接电脑的传统串口口

### 3.3 RS-232

RS-232 是一种传统串口电气标准。

常见特征：

- 电平范围和 TTL 不一样
- 可能出现正负电压
- 需要和 TTL 区分，不能直接乱接

### 3.4 USB 转串口

电脑一般没有原生串口，所以调试时常用 USB 转串口模块。

它的作用是：

- 电脑 USB 口
- 转成 TTL 串口或 RS-232 串口

常见芯片有：

- CH340
- CP2102
- PL2303
- FT232

### 3.5 最重要的一条

接线前一定先确认：

- 目标板是 `TTL` 还是 `RS-232`
- 电平是多少，`3.3V` 还是 `1.8V`
- USB 转串口模块输出的电平是否匹配

接错电平轻则没法通信，重则损坏硬件。

## 4. 串口通信需要约定哪些参数

UART 本质上是“大家约定好规则后再通信”。最核心的参数有下面这些。

### 4.1 波特率

波特率表示每秒传输多少位数据，常见值有：

- `9600`
- `115200`
- `230400`
- `460800`

嵌入式 Android 里最常见的调试波特率通常是 `115200`。

### 4.2 数据位

常见是 `7` 位或 `8` 位，开发和调试里最常见的是 `8` 位。

### 4.3 校验位

常见有：

- `N`：None，不校验
- `E`：Even，偶校验
- `O`：Odd，奇校验

初学阶段最常见组合是 `8N1`，意思是：

- 8 位数据位
- 无校验
- 1 位停止位

### 4.4 停止位

常见是 `1` 位或 `2` 位，最常见也是 `1` 位。

### 4.5 流控

流控用于控制发送速度，避免缓冲区溢出。

常见有：

- 无流控
- 软件流控
- 硬件流控

初学时建议先记住：

- 调试口很多时候不需要流控
- 设备通信时要看对端是否要求流控

---

## 5. 串口的基本引脚

最常见的 UART 至少会有这几个信号：

- `TX`：发送
- `RX`：接收
- `GND`：地

常见连线原则：

- 设备 `TX` 接对方 `RX`
- 设备 `RX` 接对方 `TX`
- 两边 `GND` 必须相连

如果你看到更多引脚，例如 `RTS`、`CTS`，那通常和硬件流控有关。

## 6. Linux 里串口长什么样

在 Linux 里，UART 通常会表现成字符设备节点，例如：

- `/dev/ttyS*`
- `/dev/ttyUSB*`
- `/dev/ttyAMA*`
- `/dev/ttyAS*`
- `/dev/ttyHS*`

不同平台命名会不一样，常见规律是：

- `ttyS*`：传统串口或通用串口
- `ttyUSB*`：USB 转串口设备
- `ttyAMA*`：ARM 平台上常见的串口命名

在嵌入式 Android 项目里，真正要关注的是：

- 设备树或板级配置里这个串口映射到了哪里
- 系统启动后对应哪个 `/dev/ttyXXX`
- 这个口被谁占用了

### 6.1 一个实战判断方法

当你在设备上看到类似下面的现象时：

- `dmesg` 里出现 `console=ttyFIQ0`
- `dmesg` 里注册了多个 `ttyS*`
- `/proc/tty/driver/serial` 能看到对应的 `mmio` 和 `irq`
- `/dev/ttyS*` 已经创建出来

通常可以先得出这几个结论：

- 系统主控制台走的是 `ttyFIQ0`，不一定是板载 `ttyS*`
- `ttyS1/2/3/...` 是内核识别到的板载 UART
- 每个串口都能从 `mmio` 地址反查到硬件实例
- 如果设备节点属主/属组是 `bluetooth:system`，它很可能被系统服务预留或使用

这类信息的价值在于：你不需要马上猜“哪个口能用”，而是先确认“哪些口存在、谁在用、是不是控制台”。

### 6.2 下一步通常怎么查

如果你想继续确认某个串口的用途，可以进一步看：

```bash
dmesg | grep -iE 'ttyS|ttyFIQ|serial|bluetooth'
cat /proc/cmdline
cat /proc/tty/driver/serial
ls -l /dev/ttyS*
```

需要时再去设备树或板级配置里找对应的 `serial@...` 节点。

## 7. 为什么“串口能看到设备节点”不代表就能用

串口在 Linux 里能不能正常用，不只是看有没有 `/dev/ttyXXX`。

还要看这些条件：

- 驱动是否加载成功
- 设备树是否把对应串口打开了
- 该串口是否被内核控制台占用
- 应用或系统服务是否已经打开了这个串口
- 当前用户有没有权限访问它

所以排查串口问题时，不能只盯着“文件在不在”，还要看“谁在用、怎么配、权限对不对”。

## 8. Linux 下常见的串口访问方式

### 8.1 直接打开设备文件

很多串口程序本质上就是打开 `/dev/ttyXXX`，然后按字节读写。

你后面会经常看到这些操作：

- 打开串口
- 设置波特率
- 设置数据位、停止位、校验位
- 读数据
- 写数据

### 8.2 使用终端工具

常见串口工具有：

- `minicom`
- `screen`
- `picocom`
- `cutecom`
- `gtkterm`

这些工具适合做最基础的连通性验证。

### 8.3 使用脚本或程序

当你需要做自动化测试、协议收发、产测工具时，通常会用：

- C/C++

## 9. 权限、占用和权限组

Linux 下访问串口经常遇到权限问题。

你要先理解这几个点：

- 普通用户不一定能直接访问 `/dev/ttyXXX`
- 设备节点有属主、属组和权限位
- 串口可能属于 `dialout`、`uucp`、`tty` 等组

常见现象：

- 打开时报 `Permission denied`
- 串口有设备节点，但程序打不开
- 设备被别的进程占用

初学时建议养成习惯：

- 先确认当前用户权限
- 再确认设备节点权限
- 再确认有没有进程占用

## 10. 串口调试最常见的问题

### 10.1 没有任何输出

可能原因：

- 线接反了
- 电平不匹配
- 波特率不对
- 串口被别的进程占用
- 目标设备根本没有向这个口输出

### 10.2 输出乱码

可能原因：

- 波特率不一致
- 数据位、校验位、停止位不一致
- 电平不稳定
- 接触不良

### 10.3 只能单向通信

可能原因：

- `TX/RX` 接反
- `GND` 没接好
- 对端没启用接收

### 10.4 一插上就复位或异常

可能原因：

- 接到了不该接的引脚
- 供电和信号混接
- 电平不匹配

## 11. 嵌入式 Android 场景里你要特别注意的点

和普通 Linux 串口开发相比，Android 设备上常多出一些限制。

### 11.1 串口可能被系统当作控制台

很多板子会把某个 UART 用作：

- bootloader console
- kernel console
- Android log 输出口

这意味着这个串口可能不是“随便就能拿来给应用用”的。

### 11.2 SELinux 可能限制访问

在 Android 系统里，即使你看得到设备节点，也不代表应用一定能访问。

你还要考虑：

- SELinux 策略
- `ueventd` 权限配置
- HAL 或 system service 的封装方式

### 11.3 串口常被封装成系统服务

在正式产品里，应用层通常不会直接随便打开串口，而是通过：

- HAL
- system service
- 本地守护进程

这样做是为了权限隔离和稳定性。

## 14. 入门时最容易犯的几个错误

- 以为所有“串口”都是同一种电平
- 把 `TX` 接到 `TX`，`RX` 接到 `RX`
- 不接 `GND`
- 波特率随便猜
- 认为 Linux 上有设备节点就一定能直接使用
- 忽略 Android 上的权限和 SELinux
